Teaching an Agent to Run a Mock Interview

A 0→1 tool built live with Claude Code, corrected in the open

What started as "help me prep some interview answers" became a persisted, live-updating mock-interview tool with its own review workflow — and, once that workflow proved itself, its own reusable Claude Code skill.

  • Claude Code
  • FastAPI + SQLite
  • Agent skill authoring
  • Vanilla JS, no framework

The problem

I was prepping for a platform-team leadership interview — the kind where the hiring manager tests whether you actually understand platform tradeoffs, not just whether you can recite a résumé. I'd already done deep research on the company and had a solid written interview-prep doc. It wasn't enough. Reading predicted questions off a page isn't the same as being asked them, and a static document can't react to what you actually say.

So I asked Claude Code to build something better: a tool where I write real answers, and Claude reviews them the way an actual interviewer would react — reads what I said, reacts to it specifically, and either signs off or pushes back.

The details below are genericized — this was built for a real, active interview process, so the specific company, role, and interview content aren't reproduced here. What's real is the build: every correction, bug, and design decision described happened exactly as written.

Building it, in the open

The interesting part isn't the finished tool — it's the sequence of corrections that got it there. Several of these were me catching Claude's mistake; a couple were Claude catching its own.

Moment 1 — the trust reset

A fun prototype, and the wrong instinct

First pass: an interactive "predicted questions" tree with sample answers written in, so the UI had something to show. It looked great. It was also a problem — the sample answers were invented, dressed up to sound plausible, and nothing marked them as fabricated.

"Where did you come up with those answers?" — the question that ended that approach.

The fix wasn't cosmetic. Every invented answer got stripped out and replaced with an empty text box. The standing rule from that point forward: Claude writes the questions (predictions of what an interviewer might ask — clearly Claude's own analysis), but never the answers. Those are real or they don't exist.

Moment 2 — making it real

From a static page to a persisted workflow

Once answers had to be genuinely mine, the tool needed real state: a SQLite-backed FastAPI service tracking each question's raw answer, Claude's cleaned-up version, review status, and a full feedback history — not just current state, an append-only log, since the correction trail turned out to matter as much as the final answer.

Every cleaned-up version starts as a draft. Nothing is final until I approve it. "Request changes" reopens it with my actual feedback attached, which Claude reads and revises against — a real edit loop, not a one-shot rewrite.

Moment 3 — making it feel alive

Live status instead of a silent black box

Early on, saving an answer and asking Claude to review it meant switching to a chat window and waiting, with no signal in the tool itself that anything was happening. Two additions fixed that: a background watcher (Claude Code's Monitor tool, polling the database) that notifies Claude the moment a new answer lands, and a "claim" step Claude calls immediately on picking it up — before the actual review is even written — so the page shows a live "reviewing now" state within seconds. The page itself polls and re-renders automatically; no manual refresh, ever.

Moment 4 — a structural bug

Question 2 was already answered by Question 1

The tool originally let the next planned question stay visible while a deep follow-up thread on the current question was still open. In practice, that meant a later question sometimes got answered by accident, mid-conversation, on an earlier one — leaving it redundant by the time it was actually reached. The fix: a question only unlocks once its entire follow-up thread is answered, and once that happens, Claude re-reads everything said so far and rewrites or cuts whatever's next if it's already been covered — rather than mechanically asking a question that's already been answered.

Moment 5 — inverting the interview

"Instead of pills, choose the question"

The tool's early design offered multiple candidate follow-up questions as clickable options — pick which direction to explore next. It was a UX pattern borrowed from a completely different kind of tool, and it quietly broke the premise: a real interviewer doesn't hand you a menu of which hard question you'd like to answer. Claude now picks the single most likely next question itself, using judgment about how the conversation would actually flow, and keeps the alternatives on file to revisit later rather than surfacing them as a choice.

Moment 6 — catching itself

Specific isn't the same as substantiated

This is the one Claude caught wrong on its own initiative being questioned, not proactively. Reviewing an answer that had good procedural detail — real specifics about process and people involved — but whose actual core claim boiled down to an unsupported "the two things were just too different to unify," Claude praised it as "strong and specific" anyway. Called on it directly, the honest answer was: no, that claim should have triggered a follow-up, and didn't, because surrounding detail elsewhere in the same answer made the whole thing feel substantiated without actually checking the one sentence the whole conclusion rested on.

The fix went into the review process itself: certain words — "complicated," "different," "aligned," "the right call" — now trigger an explicit check, every time, regardless of how solid the rest of the answer reads.

Moment 7 — scaling the interface

From one long scroll to Miller columns

As review threads got several questions deep, a single scrolling page stopped being usable. The fix borrowed a pattern from a companion tool in the same project (a research-notes dashboard already using Miller columns): sections in the first column, that section's question sequence in the second, and the full detail — answer, review, follow-ups — in the third. Later questions stay visibly locked until earlier ones are actually finished, so the structure itself communicates the "finish this before moving on" rule instead of just enforcing it silently.

Turning the workflow into a skill

Once the review process had stabilized — claim before working, a consistent formatting convention, the hand-wavy-language check, one follow-up instead of a menu, re-checking downstream questions instead of asking something already covered — it stopped being something worth re-explaining from scratch for the next company. So it became a Claude Code skill: a documented, reusable process Claude loads automatically whenever this kind of review comes up again.

The skill's checklist isn't hypothetical — it's built directly from the two real mistakes above, written down as a rationalization table and a red-flags list, so the next session starts already knowing not to repeat them instead of relying on me to catch it live a second time.

This is the part I think matters most: the corrections above didn't just fix the tool once. They got captured as process, so the system actually gets better with use instead of resetting to the same baseline every session.

What it's built on

FastAPI + SQLite backend (four tables: raw answers, reviews, reactive follow-ups, and an append-only event log). Vanilla JS frontend — no framework, no build step — polling the API every few seconds for live updates. A background Monitor-driven watcher script that turns database changes into notifications inside the coding session itself, so review work starts within seconds of an answer being saved rather than waiting to be asked. Every meaningful design correction above happened through direct conversation with Claude Code, verified live in a browser rather than assumed to work.
